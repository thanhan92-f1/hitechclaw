## 2. CI/CD Pipeline

### CI — Runs on every PR and push to main/develop

| Job | Purpose |
|-----|---------|
| **Lint** | ESLint across all packages |
| **TypeCheck** | `tsc -b` in dependency order |
| **Security Audit** | `npm audit --omit=dev` |
| **Docker Build** | Verify Dockerfiles build successfully |
| **Trivy Scan** | Container vulnerability scanning (blocks on CRITICAL) |

### CD — Image Publishing

Triggers: push to `main`, tags `v*`

- Builds `hitechclaw-server` and `hitechclaw-web` images
- Pushes to GHCR with tags: `latest`, `v2.1.0`, `sha-abc1234`
- Multi-platform: `linux/amd64` + `linux/arm64`
- GHA build cache for fast rebuilds

### Build Order (dependency graph)

```
shared → db → core → integrations → domains → ml
                  ↘ skills → skill-hub
                  ↘ channels/*
                              ↘ gateway → server
```

## 3. Environments

| Environment | Branch | Compose Command | DB Ports |
|-------------|--------|----------------|----------|
| Development | feature/* | `docker compose up --build` | Exposed |
| Staging | develop | `docker compose -f ... -f ...prod.yml up -d` | Internal |
| Production | main (tags) | `docker compose -f ... -f ...prod.yml up -d` | Internal |

Production overrides include:
- Closed database ports (internal Docker network only)
- Resource limits (memory, CPU caps)
- Pre-built GHCR images instead of local build
- PostgreSQL tuning (shared_buffers, connection limits)
- JSON log rotation (50MB × 5 files)

## 4. Health Checks

| Endpoint | Type | Checks |
|----------|------|--------|
| `GET /health` | Liveness | App is running (uptime, version) |
| `GET /health/ready` | Readiness | PostgreSQL + MongoDB connections alive |
| `GET /health/deep` | Deep | PG latency, Mongo latency, memory usage |

Deep health response:
```json
{
  "status": "ok | degraded | unhealthy",
  "version": "2.1.0",
  "uptime": 86400,
  "checks": {
    "postgres": { "status": "ok", "latency_ms": 2 },
    "mongodb": { "status": "ok", "latency_ms": 5 },
    "memory": { "status": "ok", "latency_ms": 0 }
  },
  "memory": { "used_mb": 256, "heap_mb": 180 }
}
```

## 5. Database Operations

### Migrations
- Auto-run on server startup via `runMigrations()`
- Forward-only (revert = new migration that undoes changes)
- Always test on staging before production

### Backups

| Database | Script | Default Output | Retention |
|----------|--------|---------------|-----------|
| PostgreSQL | `deploy/scripts/backup-postgres.sh` | `./backups/postgres/` | 30 days |
| MongoDB | `deploy/scripts/backup-mongodb.sh` | `./backups/mongodb/` | 30 days |
| Redis | Not backed up | — | Cache only |

### Restore

```bash
# PostgreSQL
pg_restore -h $PG_HOST -U hitechclaw -d hitechclaw < backup.dump

# MongoDB
mongorestore --uri="$MONGODB_URL" --gzip --drop dump/
```

## 6. Security

### Container Security
- Non-root user `hitechclaw` (UID 1001) in server Dockerfile
- Trivy scan in CI (blocks on CRITICAL vulnerabilities)
- `.dockerignore` excludes node_modules, .git, .env
- Base images updated via Dependabot

### Network Security
- Database ports closed in production (Docker internal network)
- Nginx handles SSL termination
- CORS restricted to configured origins
- Security headers in Nginx (cache-control, immutable assets)

### Secret Management
- `.env` files NEVER committed to git
- `JWT_SECRET` must be ≥ 64 chars random (`openssl rand -hex 32`)
- Production env template: `deploy/env/.env.production.example`
- API keys rotated every 90 days

## 10. SLO / SLI / Error Budget

### Service Level Indicators (SLIs)

| SLI | Measurement |
|-----|-------------|
| **Availability** | % requests returning 2xx/3xx over 5-min window |
| **Latency** | p95 response time for `POST /api/chat` |
| **Error rate** | % requests returning 5xx |
| **Durability** | % successful DB writes confirmed readable |

### Service Level Objectives (SLOs)

| Service | SLO | Error Budget (30 days) |
|---------|-----|----------------------|
| API availability | 99.5% | 3h 36m downtime |
| p95 latency < 1s | 95% of requests | 5% allowed to exceed |
| 5xx error rate < 0.5% | 99.5% clean requests | — |

### Error Budget Policy

- **Budget > 50% remaining** → Normal velocity, experiments allowed
- **Budget 10–50% remaining** → Slow down risky deploys, focus on reliability
- **Budget < 10% remaining** → Freeze non-critical deploys, reliability work only
- **Budget exhausted** → Incident review required before resuming deploys

## 12. Alerting & On-Call

### Alert Severity Matrix

| Severity | Definition | Response Time | Notify |
|----------|-----------|---------------|--------|
| **SEV1** | Total outage, data loss risk | 15 min | On-call + manager |
| **SEV2** | Major feature broken, > 10% users impacted | 30 min | On-call |
| **SEV3** | Degraded performance, workaround exists | 4 hours | Slack #alerts |
| **SEV4** | Minor issue, no user impact | Next business day | Ticket only |

### Alert Rules

```yaml
# Example Prometheus rules
- alert: HighErrorRate
  expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.05
  for: 2m
  severity: SEV2

- alert: HighLatency
  expr: histogram_quantile(0.95, http_request_duration_seconds) > 2
  for: 5m
  severity: SEV3

- alert: DatabaseDown
  expr: up{job="postgres"} == 0
  for: 1m
  severity: SEV1
```

### On-Call Rotation

- Primary on-call: 1-week rotation
- Escalation: primary → secondary (15 min) → engineering lead (30 min)
- Acknowledge SLA: SEV1 within 15 min, SEV2 within 30 min
- Use `#incidents` Slack channel for coordination

## 13. Deployment Strategies

### Current: Rolling Deploy (default)
Zero-downtime rolling restart via Docker Compose. Suitable for most releases.

### Blue-Green Deploy (breaking changes)

```bash
# 1. Deploy to "green" stack alongside "blue"
docker compose -f docker-compose.prod.yml -p hitechclaw-green up -d

# 2. Smoke test green
./deploy/scripts/health-check.sh https://green.hitechclaw.internal

# 3. Switch Nginx upstream to green
nginx -s reload

# 4. Keep blue running for 30 min (fast rollback window)
# 5. Stop blue after confirmation
docker compose -p hitechclaw-blue down
```

### Canary Deploy (high-risk changes)

Route 10% of traffic to new version via Nginx:

```nginx
upstream hitechclaw_backend {
  server hitechclaw-blue:3000 weight=9;   # 90% existing
  server hitechclaw-green:3000 weight=1;  # 10% canary
}
```

Monitor for 15 min → if error rate unchanged → full cutover.

### Pre-Deploy Checklist

- [ ] All CI checks green
- [ ] Staging deploy verified
- [ ] Migration script reviewed (no destructive changes)
- [ ] Rollback plan documented
- [ ] On-call notified for SEV1-risk deploys

## 15. Secret & Certificate Rotation

### Secret Rotation Schedule

| Secret | Rotation | Method |
|--------|----------|--------|
| `JWT_SECRET` | Every 90 days | `openssl rand -hex 32`, rolling restart |
| Database passwords | Every 90 days | Update `.env`, restart services |
| LLM API keys | Every 90 days | Rotate in provider console, update env |
| OAuth client secrets | Every 180 days | Rotate in provider console |

### Rotation Procedure

```bash
# 1. Generate new secret
NEW_SECRET=$(openssl rand -hex 32)

# 2. Update production env (coordinated with team)
# Edit deploy/env/.env.production

# 3. Rolling restart (zero-downtime)
docker compose -f docker-compose.prod.yml up -d --no-deps hitechclaw

# 4. Verify health
./deploy/scripts/health-check.sh
```

### SSL Certificate Renewal

Certificates auto-renewed via Certbot (Let's Encrypt). Manual check:

```bash
# Check expiry
openssl x509 -enddate -noout -in /etc/letsencrypt/live/hitechclaw.io/cert.pem

# Force renewal
certbot renew --force-renewal
```

Alert 30 days before expiry.

## 16. Backup Verification

**Rule: An untested backup is not a backup.**

### Monthly Restore Test

```bash
# PostgreSQL — restore to test container
docker run --rm -e POSTGRES_PASSWORD=test postgres:18 \
  pg_restore --verbose -d hitechclaw_test < latest.dump

# MongoDB — restore to test container
docker run --rm mongo:7 \
  mongorestore --uri="mongodb://localhost:27017" --gzip --drop dump/
```

Verify row/document counts match source. Document results in `deploy/runbooks/disaster-recovery.md`.

### Backup Checklist

- [ ] Both PG and Mongo backups complete daily
- [ ] Backups transferred to off-site storage (S3/GCS)
- [ ] Restore test performed monthly
- [ ] Backup alerts configured (notify if backup job fails)
- [ ] Retention policy enforced (30 days local, 90 days remote)

## 17. Post-Mortem Process

Required for all SEV1 incidents and optional for SEV2.

### Timeline

| When | Action |
|------|--------|
| During incident | Designate incident commander, update #incidents every 15 min |
| Within 24h | Draft post-mortem document |
| Within 72h | Team review, finalize action items |
| Within 2 weeks | All P1 action items resolved |

### Post-Mortem Template

```markdown
## Incident Summary
- **Date**: 
- **Severity**: SEV1/SEV2
- **Duration**: X hours Y minutes
- **Impact**: N users affected, N% requests failed

## Timeline
- HH:MM — Event description

## Root Cause
[Technical explanation]

## Contributing Factors
- Factor 1
- Factor 2

## Action Items
| Item | Owner | Due Date | Priority |
|------|-------|----------|----------|
| Fix X | @dev | YYYY-MM-DD | P1 |

## Lessons Learned
```

**Blameless culture**: Post-mortems focus on systems and processes, not individuals.
